# Method Map for Information Analysis

Use this map to choose the smallest method that fixes the weak point of the analysis. Do not run all methods by default.

Core cycle:

```text
question → context → sources → facts → hypotheses → checks → alternatives → uncertainty → conclusion → decision → revision
```

## 1. Analytical question

Use when the task is a topic, request, or vague concern.

Procedure:

1. Write the topic as given.
2. Name the decision, explanation, forecast, comparison, or description needed.
3. Name audience and use.
4. Set boundaries: object, period, segment, exclusions.
5. Write one neutral question.
6. Check that it does not contain a preferred answer.

Failure prevented: collecting everything around a topic and never answering anything.

## 2. Source map

Use when information quality is uneven or sources may be biased, stale, dependent, or partial.

Procedure:

1. List sources separately from claims.
2. For each source, write what it literally says.
3. Check proximity to primary data.
4. Check freshness, completeness, independence, incentives, and verifiability.
5. State what the source can support and what it cannot prove.
6. Assign source role: ground, signal, context, hypothesis, weak hint.

Failure prevented: treating authority, volume, or repeated citation as proof.

## 3. Fact / interpretation / hypothesis split

Use when a text sounds convincing but the reasoning jump is unclear.

Procedure:

1. Break the text into claims.
2. Mark each as observation, data, fact, interpretation, assumption, hypothesis, conclusion, or recommendation.
3. For each fact, name source and limit.
4. For each interpretation/hypothesis, name what would test it.
5. Rewrite the conclusion so its strength matches its grounds.

Failure prevented: smuggling interpretation into “facts”.

## 4. Map of unknowns

Use when data is incomplete, contradictory, or confidence is too high.

Procedure:

1. Write exact unknowns, not “lack of data”.
2. For each unknown, say which conclusion or decision depends on it.
3. Judge importance: changes answer/action or only adds detail.
4. Name source/check, effort, and deadline.
5. Decide what to do if it cannot be known.
6. Adjust confidence.

Failure prevented: waiting for impossible certainty or ignoring critical gaps.

## 5. Hypothesis generation

Use when there is a situation but the cause/mechanism/answer is unclear.

Procedure:

1. Describe the observed pattern without explaining it.
2. Write the first obvious version.
3. Add at least two alternatives with different mechanisms.
4. For each hypothesis, write expected signs.
5. For each, write what would weaken it.
6. Pick the first check that distinguishes versions.

Failure prevented: falling in love with the first plausible explanation.

## 6. Hypothesis / evidence matrix

Use when several explanations compete.

Procedure:

1. List 3–5 hypotheses.
2. Create one shared evidence list.
3. Mark each evidence item against each hypothesis: +, -, 0, ≈, ?.
4. Highlight diagnostic evidence.
5. Highlight contradictions and absent expected signs.
6. Revise hypotheses.
7. Write provisional conclusion and next check.

Failure prevented: counting confirmations instead of comparing explanations.

## 7. Key assumption check

Use before a recommendation, forecast, scenario, launch, or strong conclusion.

Procedure:

1. Take one conclusion/recommendation.
2. Rewrite it as: if X is true, Y is justified.
3. List assumptions required for Y.
4. Separate load-bearing assumptions from background assumptions.
5. Rate evidence and checkability for each load-bearing assumption.
6. Write what changes if the assumption fails.
7. Rewrite the conclusion with applicability and revision conditions.

Failure prevented: hiding fragile assumptions under confident recommendations.

## 8. Indicators and signals

Use when the situation will evolve or the conclusion must be monitored.

Procedure:

1. Choose hypothesis/conclusion/scenario to monitor.
2. Name the mechanism.
3. Define expected signs, source, threshold, and timeframe.
4. State what strengthens, weakens, or overturns the conclusion.
5. Note false signals and alternative explanations.
6. Link each signal to action.

Failure prevented: noticing events without knowing whether they matter.

## 9. Scenario analysis

Use when a single forecast is fragile and planning must handle several plausible futures.

Procedure:

1. Name the decision that needs scenarios.
2. List factors that can change the outcome.
3. Select 2–3 key uncertainties.
4. Build 3–4 plausible scenarios with mechanisms.
5. For each, name early indicators.
6. Separate actions robust across scenarios from conditional actions.
7. Set revision points.

Failure prevented: presenting wishes or fears as scenarios.

## 10. Red team

Use when a conclusion looks persuasive and needs attack.

Procedure:

1. Write current conclusion in one paragraph.
2. Identify 3–5 load-bearing grounds.
3. For each, ask how it could be wrong.
4. Build the strongest opposing argument.
5. Find best alternative explanation.
6. Check sources, assumptions, and implications.
7. Change conclusion, confidence, monitoring, or recommendation.

Failure prevented: defending a conclusion instead of testing it.

## 11. Premortem

Use before a decision that could fail expensively.

Procedure:

1. Assume the decision failed.
2. Write the failure headline.
3. List plausible causes of failure.
4. Mark which causes are already visible as weak signals.
5. Name preventable causes.
6. Add checks, safeguards, or stop conditions.
7. Decide whether the recommendation changes.

Failure prevented: treating implementation risk as an afterthought.

## 12. Synthesis

Use when there are many pieces and the answer is scattered.

Procedure:

1. Restate the question.
2. Group facts and signals by the claim they support.
3. Remove weak or duplicate grounds.
4. Name best answer and best alternative.
5. Name confidence and why not higher.
6. State practical implication.
7. State revision condition.

Failure prevented: a report that contains material but no answer.

## 13. Analytical writing

Use when communicating analysis to decision-makers.

Procedure:

1. Put short answer early.
2. State confidence early.
3. Give 3–5 strongest grounds.
4. Name strongest alternative and why it is weaker.
5. Name uncertainty and what would change the conclusion.
6. Translate conclusion into recommended action or next check.

Failure prevented: burying the answer under process narrative.

## 14. Update protocol

Use when new information arrives.

Procedure:

1. State previous conclusion and confidence.
2. State new information and source quality.
3. Decide which hypothesis/assumption it affects.
4. Change confidence or conclusion only as much as the evidence justifies.
5. Record what remains unchanged.
6. Set next observation/check.

Failure prevented: either clinging to old conclusions or overreacting to fresh data.

## 15. Error review

Use after a wrong analysis or decision.

Procedure:

1. Reconstruct what was known then, not what is known now.
2. Mark facts, assumptions, hypotheses, and conclusions.
3. Identify missed signals and over-weighted signals.
4. Identify missing alternatives.
5. Identify process failure: question, source, evidence, assumption, confidence, communication, revision.
6. Extract one reusable rule.

Failure prevented: hindsight blame instead of learning.
